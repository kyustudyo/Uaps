🚀 URTS Scripts Architecture

Unity RTS 게임의 스크립트 구조와 상호작용 관계 분석

📋 프로젝트 개요

이 문서는 Unity RTS(Real-Time Strategy) 게임의 스크립트 아키텍처를 분석합니다. Event Bus 패턴을 중심으로 한 컴포넌트 간 상호작용과 모듈화된 설계를 다룹니다.

주요 특징:
  • 🎯 Event Bus 패턴으로 느슨한 결합 구현
  • 🤖 모듈화된 로봇 파츠 시스템
  • 🎮 직관적인 플레이어 입력 처리
  • 📷 동적 카메라 제어

Event Bus 시스템

namespace GameDevTV.RTS.EventBus { public static class Bus<T> where T : IEvent { public delegate void Event(T args); public static event Event OnEvent; public static void Raise(T evt) { OnEvent?.Invoke(evt); } } }

유닛 선택 이벤트 발행

public void Select() { // 시각적 피드백 if (decalProjector != null) { decalProjector.gameObject.SetActive(true); } // 이벤트 발행 Bus<UnitSelectedEvent>.Raise(new UnitSelectedEvent(this)); }

이벤트 구독 및 처리

private void Awake() { // 이벤트 구독 Bus<UnitSelectedEvent>.OnEvent += HandleUnitSelected; } private void HandleUnitSelected(UnitSelectedEvent evt) { // 기존 선택 해제 if (selectedUnit != null) { selectedUnit.Deselect(); } // 새 유닛 선택 selectedUnit = evt.Unit; }

🎯 설계 원칙

1. 느슨한 결합 (Loose Coupling)

Event Bus 패턴을 통해 컴포넌트 간 직접 참조를 최소화하고, 이벤트를 통한 통신으로 유연성을 확보했습니다.

2. 단일 책임 원칙 (SRP)

각 클래스는 명확한 단일 책임을 가집니다:

  • PlayerInput: 입력 처리 및 카메라 제어
  • Worker: 기본 유닛 동작
  • WorkerThreeRobot: 복합 로봇 동작
  • RobotPart: 개별 파츠 회전

3. 인터페이스 분리 원칙 (ISP)

ISelectable과 IMoveable 인터페이스로 기능을 분리하여 필요한 기능만 구현할 수 있도록 했습니다.

4. 개방-폐쇄 원칙 (OCP)

새로운 유닛 타입을 추가할 때 기존 코드 수정 없이 인터페이스만 구현하면 됩니다.

⚠️ 주의사항 및 개선점

메모리 누수 방지: OnDestroy()에서 이벤트 구독을 해제해야 합니다.
레이어 충돌 매트릭스: World Bounds와 Floor 레이어 충돌을 비활성화해야 정상 작동합니다.
확장 가능성:
  • 새로운 이벤트 타입 추가 (UnitMovedEvent, UnitAttackedEvent 등)
  • 다양한 로봇 파츠 타입 지원
  • AI 시스템과의 통합

🎉 결론

이 URTS 프로젝트는 Event Bus 패턴을 중심으로 한 잘 설계된 아키텍처를 보여줍니다. 모듈화된 구조와 명확한 책임 분리로 유지보수성과 확장성을 확보했으며, 특히 Nova 1492 스타일의 로봇 파츠 시스템은 독창적인 구현입니다.

이 문서를 통해 새로운 개발자도 프로젝트의 전체적인 구조와 각 컴포넌트의 역할을 쉽게 이해할 수 있을 것입니다.
